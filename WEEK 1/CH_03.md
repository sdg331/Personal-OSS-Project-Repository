## 2. 깃(Git) 개요와 특징

- 깃 정의 및 역사  
  - 분산 버전 관리 시스템(Distributed Version Control System, DVCS)  
  - 2005년 리누스 토발즈(Linus Torvalds)가 리눅스 커널 개발을 위해 설계  
  - 현재는 주니오 하마노(Junio Hamano)가 유지보수 담당  
  - 대표적인 오픈소스 소프트웨어 프로젝트로 Git 자체도 GitHub와 git.kernel.org에 저장소가 공개됨[1]

- 깃 사용의 장점  
  - 모든 개발자가 자신의 지역 시스템(Local)에 **코드 전체 사본**을 보유  
  - 네트워크가 없어도 로컬 저장소에서 대부분의 작업(커밋, 브랜치 생성 등) 수행 가능  
  - 소스 코드의 모든 변경 사항이 커밋 이력으로 남아 다른 사용자가 추적 가능  
  - 분산 구조로 중앙 서버 장애 시에도 각 로컬 저장소가 백업 역할을 수행[1]

- 깃의 주요 기능  
  - 파일 변경 이력을 시간에 따라 추적하는 버전 관리 시스템  
  - 여러 개발자가 동시에 같은 프로젝트에서 작업할 수 있도록 지원  
  - 브랜치를 통한 병행 개발(Parallel, Non-linear Development) 지원  
  - 분산 버전 제어 도구로서 각 개발자 PC에 전체 이력 저장[1]

***

## 3. 깃 설치 시 제공 소프트웨어: Git Bash, Git GUI

- Git Bash (CLI, Command Line Interface)  
  - 텍스트 기반 명령 행 인터페이스  
  - 처음에는 진입 장벽이 있으나, 한 번 익히면 모든 기능을 가장 유연하게 사용할 수 있음  
  - Mac의 Terminal, Windows의 CMD·PowerShell과 유사한 역할  
  - CLI를 사용할 줄 알면 GUI 사용도 가능하지만, 그 반대는 성립하지 않는다는 점이 강조됨[1]

- Git GUI (Graphical User Interface)  
  - 그래픽 기반의 깃 클라이언트  
  - Git 기능 중 일부만 GUI로 구현하여 상대적으로 단순한 편  
  - 기본적인 작업(저장소 생성, 클론, 최근 저장소 열기 등)을 메뉴 기반으로 제공[1]

***

## 4. 깃 내부 저장소 주요 영역 구조

- 작업 디렉토리(Working Directory, Working Folder)  
- 작업 공간(Work Space), 작업 트리(Working Tree)  
  - 실제 파일이 있는 디렉토리  
  - Git이 아직 추적하지 않는 파일: untracked  
  - Git이 추적 중이며 수정된 파일: modified[1]

- 스테이징 영역(Staging Area, Stage Area, Index)  
  - 커밋 대상이 될 변경 내용을 임시로 모아두는 영역  
  - 상태: staged, indexed[1]

- 깃 저장소(Git Repository, .git Directory)  
  - 커밋된 이력 데이터가 저장되는 내부 데이터베이스  
  - 상태: committed[1]

- 임시 저장소(Stash)  
  - 아직 커밋하고 싶지 않은 변경 사항을 임시로 숨겨두는 공간  
  - 상태: stash[1]

***

## 5. 깃에서 상태 간의 이동 명령

- 작업 흐름 기본 구조  
  - Work space → Staging Area(Index) → Local Repository → Remote Repository[1]

- add  
  - 작업 디렉토리(Work space)에서 변경된 파일을 스테이징 영역(Index)으로 이동시키는 명령  
  - 예: `git add 파일명`[1]

- commit  
  - 스테이징 영역(Index)에 있는 변경 사항을 로컬 저장소(Local Repository)에 영구적으로 기록  
  - 예: `git commit -m "메시지"`[1]

- push  
  - 로컬 저장소(Local Repository)의 커밋들을 원격 저장소(Remote Repository)로 업로드  
  - 팀원과 변경 사항을 공유하는 단계[1]

- pull  
  - 원격 저장소(Remote Repository)에 있는 최신 변경 사항을 로컬 저장소/작업 디렉토리로 가져와 병합  
  - 외부 변경 내용을 내 작업 환경에 반영하는 단계[1]

## 6. 브랜치(Branch) 개념

- 기본 개념  
  - 나무의 가지(branch), 지점, 분기와 같은 의미  
  - 기존 작업 흐름에서 갈라져 나온 또 다른 독립적인 작업 흐름(Workflow)을 의미[1]

- 실생활 비유  
  - 여러 파일을 이름만 바꾸어 복사하면서 버전을 관리하는 방식과 유사  
  - 특정 시점의 파일 집합을 통째로 새로운 폴더에 복사해 다른 방향으로 수정해 나가는 것과 비슷한 개념[1]

- 비선형 개발(Non-linear Development)  
  - 브랜치를 활용하면 메인 라인과는 다른 기능 개발을 동시에 진행 가능  
  - 필요 시 브랜치 간 병합(Merge)을 통해 변경 사항을 하나의 흐름으로 합칠 수 있음[1]

## 7. 깃허브(GitHub) 이해

- Git과 GitHub의 구분  
  - Git: 분산 버전 제어 소프트웨어(DVCS)  
  - GitHub: Git 저장소를 위한 웹 기반 호스팅 서비스이자 협업 플랫폼  
  - Git은 로컬 도구, GitHub는 온라인 서비스로 이해하면 됨[1]

- 깃허브 소개  
  - 버전 관리를 위한 서버 저장소 및 프로젝트 개발을 위한 협업 관리 서비스  
  - 시스템 개발자와 운영 담당자(IT 전문가) 간 소통, 협업, 통합 및 자동화를 지원  
  - 전 세계 개발자가 프로젝트 소스를 공유하고 협업할 수 있는 플랫폼  
  - 개발자 소셜 네트워크(SNS) 기능도 제공[1]

- 깃허브 서비스 특징  
  - Git 기반의 글로벌 웹 호스팅 서비스  
  - 세계 최대 규모의 오픈소스 공유 플랫폼  
  - 공개/비공개 저장소, 코드 리뷰, 문서화, 커뮤니티 기능 등 제공  
  - 무료 요금제에서는 공개 저장소 중심 사용, 비공개 저장소는 과거에는 유료 중심이었음[1]

- 마이크로소프트 인수  
  - 2018년 6월 4일, 마이크로소프트가 약 75억 달러에 GitHub 인수 발표  
  - 2018년 8월 기준, 9600만 개 이상의 프로젝트, 3100만 명 이상의 개발자 참여 플랫폼로 성장[1]

## 8. 정리 요약

- Git  
  - 시간 흐름에 따른 파일·프로젝트 변경 내역을 추적하는 분산 버전 관리 시스템  
  - Working Directory, Staging Area, Repository, Stash로 내부 구조 구성[1]

- GitHub  
  - Git 저장소를 온라인에서 호스팅하여 코드 공유와 협업을 쉽게 만드는 웹 서비스  
  - 전 세계 개발자들이 오픈소스 및 사내 프로젝트를 함께 개발하는 협업 플랫폼[1]

- 브랜치  
  - 파일 집합 전체를 복사해 별도의 버전으로 독립 개발을 진행하는 개념  
  - 다양한 기능 개발을 병렬적으로 수행하고, 병합을 통해 한 프로젝트로 통합 가능[1]

- 깃 + 깃허브  
  - 오픈소스 소프트웨어 및 일반 소프트웨어 개발에서 사실상 표준 조합으로 활용  
  - 버전 관리와 협업 개발을 동시에 지원하는 필수 도구 스택으로 자리 잡음[1]
