# 19. 오픈소스 소프트웨어 개요 (Lecture Notes)

## 1. 오픈소스 소프트웨어(OSS) 개요
### [cite_start]1.1 정의 및 특징 [cite: 301, 302, 303]
* **OSS (Open Source Software)**: 누구나 특별한 제한 없이 코드를 보고 사용할 수 있는 오픈소스 라이선스를 만족하는 SW.
* **핵심**: 소스코드 공개, 복제 가능, 수정 및 재배포 허용.
* [cite_start]**OSI (Open Source Initiative)**: 오픈소스 정의(OSD) 관리 및 인증 마크 부여 기관[cite: 305, 307].
* [cite_start]**의의**: 여러 사람이 버그를 개선하고 발전시켜 소프트웨어 품질 향상 속도가 빠름[cite: 312].

### [cite_start]1.2 자유 소프트웨어(Free Software)와의 관계 [cite: 322, 323, 329]
* **자유 소프트웨어**: 리처드 스톨먼(Richard Stallman)과 FSF(자유소프트웨어재단) 주도.
* **GNU 프로젝트**: "GNU is Not Unix" (유닉스와 호환되지만 자유로운 OS).
* **철학**: '무료(Free of charge)'가 아닌 **'자유(Freedom)'**를 강조.
* **카피레프트(Copyleft)**: 저작권(Copyright)의 반대 개념. 저작권을 인정하되, 누구나 복제/사용할 권리를 보장함.
* [cite_start]**OSS와의 차이**: 자유 소프트웨어는 '자유'라는 이념을 중시, OSS는 소스 코드의 형태와 실용성을 중시[cite: 344].

## 2. 오픈소스 개발 및 협업
### [cite_start]2.1 개발 지원 플랫폼 [cite: 356]
* 소스코드 호스팅 및 협업 도구: **GitHub, GitLab, Bitbucket** 등.
* 여러 개발자가 소스코드를 공유하고 협업할 수 있는 환경 제공.

### [cite_start]2.2 협업 방식 및 효과 [cite: 361, 363]
* 커뮤니티 내 동료 평가(Peer Review)에 의존.
* 인사이트와 아이디어 공유를 통해 혁신적인 솔루션 도출.
* 재활용성과 접근성을 높여 지속 가능한 SW 개발 생태계 구축.

## 3. 오픈소스 소프트웨어의 장단점
### [cite_start]3.1 장점 [cite: 371, 375, 376]
* **교육 및 연구**: 소스코드가 공개되어 있어 프로그래밍 기술 습득 및 실습 용이.
* **커스터마이징**: 기업이나 사용자의 요구 사항에 맞춰 수정 가능.
* **혁신 지원**: 기존 코드에 새로운 사용자 경험을 더해 빠른 혁신 가능.

### [cite_start]3.2 단점 [cite: 378, 380]
* **공개 의무**: 라이선스에 따라 소스코드 공개가 강제될 수 있음.
* **유지보수 어려움**: 공식적인 품질 보증이나 기술 지원을 받기 어려울 수 있음(보안 문제 등).

## 4. 주요 오픈소스 모델 및 종류
### [cite_start]4.1 LAMP 스택 (웹 서비스 모델) [cite: 388, 389, 390, 391, 392]

### [cite_start]4.2 분야별 대표 소프트웨어 [cite: 403, 419, 420, 254]
* **AI/빅데이터**: Python, Scikit-learn, Tensorflow, Pytorch.
* **모바일/웹**: Android, Mozilla Firefox.
* **오피스/기타**: LibreOffice, Git (버전 관리).

## 5. 오픈소스 라이선스 (핵심 중요!)
[cite_start]*라이선스 위반 시 법적 책임이 따르므로 각 특징을 명확히 구분해야 함.* [cite: 250, 251]

### 5.1 라이선스 유형별 상세 정리
| 라이선스명 | 특징 및 의무사항 | 대표 적용 사례 |
| :--- | :--- | :--- |
| **GPL (v2.0, v3.0)** | - **가장 강력한 카피레프트**.<br>- 사용 시 소스코드 공개 의무.<br>- 2차 저작물도 반드시 GPL로 배포해야 함(전염성). | [cite_start]리눅스 커널, Git, 마리아DB, 워드프레스 [cite: 251] |
| **AGPL** | - 네트워크로 통신하는 SW(웹 서비스 등)도 소스 공개 의무 대상에 포함.<br>- 서버에서 실행되는 서비스의 소스도 다운로드 가능해야 함. | [cite_start]MongoDB [cite: 252, 256] |
| **LGPL** | - GPL보다 완화된 라이선스.<br>- 라이브러리로 동적 링크(Link)하여 사용할 경우, **전체 소스 공개 의무 없음**.<br>- 해당 라이브러리 소스만 공개하면 됨. | [cite_start]파이어폭스, 각종 라이브러리 [cite: 253, 257] |
| **Apache License** | - **소스코드 공개 의무 없음**.<br>- 아파치 재단 SW임을 명시하고 라이선스 사본 포함해야 함.<br>- 수정 배포 시 특허권 문제에서 자유로움. | [cite_start]안드로이드, 하둡 [cite: 258] |
| **MIT License** | - **가장 느슨하고 제한이 적음**.<br>- 저작권 고지만 유지하면 수정/배포/상용화 자유.<br>- 소스코드 공개 의무 없음. [cite_start]| jQuery, Bootstrap, React 등 [cite: 258] |
| **BSD License** | - MIT와 유사하게 제약이 거의 없음.<br>- 소스코드 공개 의무 없음. | [cite_start]Nginx 등 [cite: 264] |

### [cite_start]5.2 의무 강도에 따른 분류 [cite: 264, 265]
1.  **Strong Copyleft (소스 공개 필수)**: GPL, AGPL
2.  **Weak Copyleft (조건부 공개)**: LGPL, MPL
3.  **Permissive (공개 의무 없음)**: Apache, MIT, BSD

> **요약**: 의무 강도는 **GPL > LGPL > Apache/BSD/MIT** 순으로 낮아짐.

## [cite_start]6. 요약 (Summary) [cite: 260]
* 오픈소스는 소스코드를 자유롭게 읽고, 수정, 재배포할 수 있는 SW이다.
* 깃허브(GitHub) 등을 통해 협업하며 발전한다.
* 주요 라이선스(GPL, Apache, MIT 등)의 **소스코드 공개 의무 여부**를 반드시 이해해야 한다.
